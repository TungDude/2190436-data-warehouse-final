§ 01 · 4-step dimensional design

Kimball's process, step by step.

Each step asks one question — and the project answers it before moving on to the next.
01
business process
What activity are we measuring?

A business process is a low-level activity performed by the organisation — not a department or function.Kimball · Lecture 02 · slide 7

How we answered it
  • Reporting a crime incidentprimary process · ≈ 1.5 M rows from CPD's CLEAR system
  • Arresting / charging a personsecond process · enables a non-trivial bus matrix
  • Snapshots are not processessocio-economics & weather feed dimensions, not facts
two business processes one fact per process
02
grain
What does one fact row mean?

Atomic grain refers to the lowest level that data is captured in the business process — preferably the most atomic level possible.Kimball · Lecture 02 · slide 9

How we answered it
  • fact_crimeone row per reported crime incident · natural key = Chicago id
  • fact_arrestone row per arrest event · natural key = cb_no
  • Type — transaction factinsert-only; corrections upsert by id, raw S3 zone preserves audit
  • No aggregate factroll-ups are computed from the atomic grain at query time
transaction fact atomic grain idempotent upsert
03
dimensions
What surrounds the event?

Who, what, when, why, where, and how associated with the event? Add a dimension only if it takes one value per primary-dim combination.Kimball · Lecture 02 · slide 11

How we answered it · 6 dims + degenerate
  • dim_date · dim_time_of_daywhen · role-played twice · separate keeps row count tiny
  • dim_locationwhere · community · district · ward · beat · SCD2 socio
  • dim_crime_typewhat · IUCR · primary type · FBI code · index/non-index
  • dim_weather · dim_crime_flagsunder what · how · junk dim of arrest + domestic
  • case_numberdegenerate dim — kept on fact, no separate table
role-playing junk dim degenerate dim separate date / time
04
facts
What measures are true to the grain?

Facts must be true to the grain. Percentages and ratios are non-additive — store numerator and denominator instead.Kimball · Lecture 02 · slide 13

How we answered it · measures on fact_crime
  • incident_count = 1additive · trivial SUM roll-ups, no COUNT(*) outer-join surprises
  • is_arrest · is_domesticsemi-additive · SMALLINT 0/1 — SUM gives a count
  • temp_c · precip_mm · hours_to_updateadditive numeric measures
  • lat · lngnon-additive · viz only, never SUM-ed
  • No stored ratiosarrest_rate computed at query time from num / denom
additive semi-additive non-additive no stored ratios
step 0 · gather requirements · 6 acceptance questions → step 5 · finish details · surrogate keys, smart keys, unknown rows

§ 04 · Slowly changing dimensions

Every dimension implements an SCD. Type chosen by intent.

The rubric requires every dimension to implement an SCD — pick the right type per intent.
0
retain

Set once at load time. Calendar, time-of-day, hourly weather — these don't have a "change" story.

  • dim_datecalendar attributes are static
  • dim_time_of_day24 fixed rows by design
  • dim_weatherimmutable hourly observations
1
overwrite

Replace in place when the source changes. No history kept — the past is gone.

  • dim_crime_flagsjunk dim, history not needed
  • dim_arresteenot "owned" across time
2
add row

Add a new row, end-date the old. Historical facts join the version that was active when the event happened.

  • dim_crime_typeIUCR codes evolve · keep prior reports correct
2+1
hybrid

Some columns version (Type 2), others overwrite (Type 1) — choose per attribute, not per dimension.

  • dim_locationsocio attrs versioned · names overwrite
