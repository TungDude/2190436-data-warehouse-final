§ 02 · Medallion zoning

Bronze, silver, gold — three quality tiers, one bucket.

Lakehouse design pattern · Bronze raw, Silver standardised, Gold curated
Layer 01 · Lake
Bronze

Immutable landing. CSV + JSON, exactly as the publishers ship them. Append-only, partitioned by ingest date.

No transformation. Versioning on, encryption on. Re-fetches go to a new partition.

s3://…-raw/raw/<source>/ingest_date=YYYY-MM-DD/
→
Layer 02 · Lake
Silver

Type-cast, deduplicated, normalised. Parquet for columnar reads. Crawled into the Glue Catalog.

Athena-queryable. Spark writes; Athena reads; QuickSight may also point here for ad-hoc.

s3://…-raw/standardized/<source>/year=YYYY/
→
Layer 03 · Warehouse
Gold

Dimensional star schema in RDS PostgreSQL 16. SCD-aware merges, surrogate keys, conformed dimensions.

The presentation area in Kimball's component model. QuickSight talks to this.

RDS · schema dw.* (fact_crime · dim_*)

The two halves of the brief map cleanly onto the same picture: the classic Source → Staging → Presentation → Access component model on the warehouse side, and the modern S3 + Catalog + Spark + Athena lakehouse stack on the lake side. Bronze + silver are the lakehouse, gold is the warehouse, and the two coexist rather than compete.

§ 05 · Star schema

One fact at the centre. Six dimensions around it.

The grain is one row per reported incident, identified by Chicago's stable id. Two role-playing date FKs (occurrence + report) reach back to dim_date. Hourly weather and an SCD2-tracked location dimension hang off the same fact.

GrainOne reported crime incident
TypeTransaction fact, atomic grain
SurrogatesBIGINT IDENTITY · YYYYMMDD smart key
§ 06 · Bus matrix

Two business processes, one set of dimensions.

Conformed-dimension bus matrix · drives drill-across between facts
Process dim_date dim_time dim_location dim_crime_type dim_weather dim_flags dim_arrestee charge bridge
Crime Reportingfact_crime · primary ● ● ● ● ● ● · ·
Arrest Eventsfact_arrest · optional ● ● · ● · · ● ●

Conformance gives us drill-across: for the same dim_date.date_key, how many crime reports vs. arrest events? The two facts join through the conformed dimensions, never directly. The same dim_crime_type serves both the primary type on a crime row and the IUCR-coded charges on an arrest bridge — one definition of "what kind of crime", reused.

§ 07 · Slowly changing dimensions

Every dimension implements an SCD. Type chosen by intent.

The rubric requires every dimension to implement an SCD — pick the right type per intent.
dim_date
Type 0 · retain
Calendar attributes (month, quarter, holiday) never legitimately change.
dim_time_of_day
Type 0 · retain
Hour-of-day attributes are static; a 24-row dimension by design.
dim_location
Type 2 + 1 · hybrid
Socioeconomics (poverty, hardship, income) version per ACS snapshot. Static labels (community-area name) overwrite.
dim_crime_type
Type 2 · add row
When CPD retires/replaces an IUCR code we keep historical reports correct against the then-active description.
dim_weather
Type 0 · retain
Each row is an immutable hourly observation — there is no "change" concept.
dim_crime_flags
Type 1 · overwrite
Junk dimension — the cartesian product of two booleans. New combos add a row, never edit.
dim_arrestee
Type 1 · overwrite
Source provides race only, not "owned" by a person across time.
SCD3 · add column
explicitly skipped
We need the full history of socioeconomic indicators (> 3 snapshots), not just current vs. prior. Type 2 is correct.
§ 08 · The big-data component

Spark for volume.
S3 + Parquet for variety.

The rubric asks for one big-data technology with justification. We bring two — both visible in the diagram, both taught in class, both load-bearing rather than decorative.

SparkDataFrame joins, in-memory DAG execution
LakehouseS3 + Glue Catalog + Spark + Athena
ColumnarParquet on the silver tier
V₁
Volume — Spark on Glue

~ 2.5 M rows across crime, arrests and weather, joined on date + hour + community area + IUCR code. PySpark DataFrames do this in minutes; pandas would buckle and would miss the point of the brief's "big-data" half.

V₂
Variety — S3 lake + Parquet

CSV from four sources, JSON from one, all landed in the same bucket. Schema-on-read in bronze, columnar Parquet in silver. The Glue Catalog is the metastore that lets Spark and Athena read the same files without duplicating definitions.

∅
KV store · rejected

"Infrequent updates + fast lookup" describes neither this dataset nor any real query in the BI layer. Adding ElastiCache just to score points would be cargo cult.

∅
EMR · rejected

EMR runs the same Spark engine that Glue manages for us. The cluster overhead is not justified by 2.5 M rows.

§ 10 · The BI surface

Two dashboards, two audiences.

Built on QuickSight · SPICE-backed · refreshed an hour after the gold-layer job
Dashboard 01 · for the executive
Overview
  • Headline KPIs. Total reports, arrest rate, week-over-week delta.
  • Time series. Monthly crime count, 2018 → present, with COVID-era markers.
  • Choropleth. Reports per community area, recoloured by crime category.
  • Donut. Distribution of primary crime types within the selected window.

Answers Q1 + Q3 of the dimensional design's acceptance test.

Dashboard 02 · for the analyst
Detail
  • Heat map. Hour-of-day × day-of-week, faceted by crime category.
  • Weather scatter. Precipitation / temperature band vs. violent-crime hours.
  • SCD2 filter. "Indicators as of the event date" — drives the location dimension's history.
  • District trend. Arrest rate by district over time, drillable to beat.

Answers Q2 + Q4 + Q5.
